# CDR 설계 체크리스트 — 파묘 프로젝트 반영본

> 원본: CTO품질PMO팀 CDR 통합 checklist v0.5 (12장, 150+ 항목)
> 이 문서는 CDR 가이드 중 **설계도에 반영해야 하는 항목**만 추출·정리한 것입니다.
> 운영/조직/계약 등 사람이 수행해야 하는 업무는 제외하고, 코드/설정/아키텍처에서 역추적 가능한 항목 중심입니다.

---

## 분류 기준

| 구분 | 설명 |
|---|---|
| 🔵 설계 반영 | 소스코드/설정에서 역추적 → 설계도 산출물에 반영 |
| ⚪ 운영/조직 | 사람이 정의 (RACI, 계약, 정책 등) — 파묘 범위 외 |
| 🟡 하이브리드 | 코드에서 일부 추출 가능, 최종 확정은 사람 필요 |

## 1장. 서비스 기획 및 책임 구조 설계

### 🔵 설계 반영
- **[CDR 1.1.1]** 서비스 카탈로그 매핑 (서비스ID/오너) → `service-inventory.md` 활용
- **[CDR 1.1.1]** NFR 선언 (성능/보안/가용성) → 코드에서 Timeout, Rate Limit, 캐시 TTL 추출 (Phase 1 1-5)
- **[CDR 1.1.1]** 도메인 용어 일관성 → 용어 사전(Glossary) 크로스체크 (Phase 3)
- **[CDR 1.1.3]** 처리 방식 매핑 (실시간/배치/이벤트) → 코드에서 기능별 식별 (Phase 1 1-4)
- **[CDR 1.2.2]** KPI↔필드/이벤트 매핑 → 로그/이벤트에서 KPI 측정 포인트 식별 (Phase 1 1-6)

### 🟡 하이브리드
- **[CDR 1.1.2]** In-Scope/Out-of-Scope → 코드 기능 목록에서 추정, PO 확정 필요
- **[CDR 1.2.3]** SLO 측정 지표(SLI) → 코드에서 메트릭 수집 포인트 식별 가능

### ⚪ 운영/조직 (파묘 범위 외)
- Problem Statement, KPI 정의식/목표치, SLA/SLO 수치 확정
- RACI, Service Owner/Ops Owner, Incident Commander, FinOps Owner
- BIA, Risk Register, Severity 기준, Error Budget 정책

## 2장. 운영 환경 및 플랫폼 구조 설계

### 🔵 설계 반영
- **[CDR 2.1.1]** OS/런타임/미들웨어 버전 + EOL 확인 → Phase 1 1-1 코드 구조 분석에서 추출
- **[CDR 2.1.2]** 컨테이너 이미지/빌드 구조, SBOM → Dockerfile/CI 파이프라인 분석 (Phase 1 1-8)
- **[CDR 2.1.2]** Config/Secret 분리 원칙 → 설정 파일 구조 분석 (Phase 1 1-7 크리덴셜)
- **[CDR 2.1.3]** 네트워크 경로/포트/방화벽/SG/ACL → 인프라 설정 파일에서 추출 (Phase 1 1-8)
- **[CDR 2.1.3]** WAF/프록시/게이트웨이 경유 정책 → API Gateway 설정 분석 (Phase 2 2-7)
- **[CDR 2.2.1]** 오토스케일 트리거/상한/하한 → k8s HPA 설정 추출 (Phase 1 1-8)
- **[CDR 2.2.2]** LB/Ingress/헬스체크 → k8s Ingress/Service 설정 추출 (Phase 1 1-8)
- **[CDR 2.2.2]** 세션/상태 처리 전략 (Stateless/Sticky) → 코드에서 세션 저장소 식별 (Phase 1 1-7)
- **[CDR 2.3.3]** IaC/Config 관리 → Terraform/Helm/Ansible 등 IaC 파일 존재 여부 확인

### ⚪ 운영/조직 (파묘 범위 외)
- 패치 정책, 유지보수 창, Capacity 모델(트래픽 가정), Chaos Test 계획
- 리전/존 장애 대응, CSP 종속성 대체안, 업그레이드 롤백 절차
