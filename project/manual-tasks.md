# 설계도 파묘 — 작업자 수작업 항목 목록

> KIRO 자동 추출만으로는 완성할 수 없는 항목을 정리한 문서입니다.
> 작업자가 직접 수집·확인·판단해야 하는 활동을 Phase별로 정의합니다.
> 자동 추출 항목은 [extraction-checklist.md](extraction-checklist.md) 참조

---

## 수작업 유형 분류

| 유형 | 태그 | 설명 |
|---|---|---|
| 런타임 데이터 수집 | `[런타임]` | 모니터링/APM/로그 시스템에서 실제 운영 데이터를 수집 |
| 담당자 인터뷰 | `[인터뷰]` | 코드에서 확인 불가, 담당자에게 직접 확인 |
| 외부 도구 실행 | `[도구실행]` | KIRO 외부의 분석 도구를 직접 실행하여 결과 수집 |
| Stakeholder 확정 | `[확정]` | KIRO가 후보를 제시하면 PO/PM/담당자가 최종 확정 |
| 교차 검증 | `[교차검증]` | 서비스 간 산출물 일관성을 사람이 검증 |
| 문서 수집 | `[문서수집]` | 기존 설계 문서, 정책 문서, 운영 매뉴얼 등을 수집 |

## Phase 0: 준비 단계 수작업

### P0-1. 서비스 분석 우선순위 결정 `[확정]`
- 30개 서비스의 분석 순서를 결정한다
- 판단 기준:
  - 비즈니스 중요도 (매출 기여, 사용자 수)
  - 소스코드 확보 여부
  - 변경 빈도 (최근 6개월 커밋 수)
  - Stakeholder 긴급도 (CDR 일정, 감사 일정 등)
  - 서비스 규모 (코드 라인 수, 모듈 수)
- `service-inventory.md`에 우선순위 컬럼을 추가하여 기록
- 담당: PM + PO 합의

### P0-2. Repo URL 확정 및 접근 권한 확보 `[인터뷰]`
- 각 서비스 담당팀에 Bitbucket Repo URL을 확인
- 멀티 Repo 서비스의 경우 앱/서버 Repo를 모두 확인
- VPN 접속 및 Repo 읽기 권한을 확보
- `service-inventory.md`에 URL 기입, 소스 유무 갱신

### P0-3. 기존 설계 문서 수집 `[문서수집]`
- 서비스별 기존 설계 문서 존재 여부 확인:
  - 요구사항 명세서, 화면 설계서, API 명세서, DB 설계서
  - 아키텍처 문서, 운영 매뉴얼, 장애 대응 Runbook
  - Confluence/Wiki에 산재된 문서
- 수집한 문서는 Phase 3 추적성 매핑의 입력으로 사용
- 문서가 없는 서비스는 `❌ 기존 문서 없음`으로 기록

### P0-4. 분석 기준 시점(Baseline Commit) 확정 `[확정]`
- 각 서비스의 분석 대상 브랜치와 커밋을 확정
- 권장: `main` 또는 `master` 브랜치의 최신 태그 또는 특정 커밋
- 분석 중 코드 변경이 발생해도 Baseline 기준으로 분석 완료
- 분석 완료 후 Baseline 이후 변경분은 증분 분석으로 처리

## Phase 1: 소스코드 분석 중 수작업

### P1-1. 런타임 트래픽 데이터 수집 `[런타임]`
- APM/모니터링 시스템에서 수집:
  - API별 호출 빈도 (일/주/월 단위 TPS)
  - API별 평균/P95/P99 응답시간
  - 에러율 상위 API 목록
  - 피크 시간대 트래픽 패턴
- 화면별 진입 빈도 (GA/Firebase Analytics):
  - DAU 기준 화면별 PV, 이탈률, 체류 시간
- 수집 결과를 `services/{서비스명}/docs/extraction/runtime-data.md`에 기록

### P1-2. 슬로우 쿼리 / DB 성능 데이터 수집 `[런타임]` `[도구실행]`
- DB 모니터링 도구에서 슬로우 쿼리 목록 수집
- 실행 계획(EXPLAIN)이 필요한 쿼리 식별
- KIRO가 코드에서 추정한 N+1/성능 병목과 실제 DB 부하 비교

### P1-3. 분산 추적 데이터 수집 `[런타임]`
- Zipkin/Jaeger/OpenTelemetry에서 실제 서비스 간 호출 그래프 수집
- KIRO가 코드에서 추출한 Call Flow와 실제 런타임 호출 비교
- 코드에 없는 런타임 의존성(서비스 메시, 사이드카 등) 식별

### P1-4. 기술 부채 마커 검토 및 분류 `[확정]`
- KIRO가 추출한 TODO/FIXME/HACK/WORKAROUND 목록을 검토
- 각 마커의 심각도 판단: 🔴 즉시 / 🟡 계획적 / ⚪ 참고
- 담당: 개발 리드 또는 PM

### P1-5. 화면명 정규화 최종 확정 `[확정]`
- KIRO가 제시한 화면명 정규화 후보를 PO/UX 담당자가 검토
- 대표 명칭(Canonical Name) 최종 확정
- 용어 사전에 확정된 매핑 등록

### P1-6. DB 마이그레이션 이력 수집 `[도구실행]`
- Flyway/Liquibase 마이그레이션 스크립트가 있는 경우 수집
- 스키마 변경 이력에서 주요 변경점 정리

### P1-7. IaC(Infrastructure as Code) 파일 수집 `[문서수집]`
- Terraform, Helm Chart, Ansible 등 IaC 파일 확인
- 소스코드 Repo에 없는 경우 인프라팀에 별도 요청

### P1-8. 외부 연동 SLA/제약사항 확인 `[인터뷰]`
- 외부 연동 시스템별 SLA, Rate Limit, 점검 시간대, 데이터 포맷 제약사항 확인

## Phase 2: View 생성 중 수작업

### P2-1. Stakeholder 리뷰 체크포인트 `[확정]`

| 리뷰 대상 | 리뷰어 | 확인 사항 |
|---|---|---|
| 화면 카탈로그 | PO, UX센터 | 화면 누락, 화면명 정확성 |
| API 명세 | PM, 개발자 | API 누락, 스키마 정확성 |
| 비즈니스 규칙 | PO, PM | 규칙 누락/오류 |
| Use Case | PO | 시나리오 누락, 예외 경로 |
| 보안 현황 | 보안센터 | 위험 항목, 대응 우선순위 |
| 품질 현황 | 품질센터 | 테스트 Gap, 성능 병목 |

### P2-2. STB 하드웨어 스펙 수집 `[인터뷰]`
- 기종별 실제 하드웨어 스펙, 미들웨어/펌웨어 버전, 리소스 사용량

### P2-3. 운영 환경 정보 수집 `[인터뷰]`
- 인스턴스 수, 리소스 할당, LB/CDN 구성, 환경별 차이, 모니터링 대시보드 구성

---

## Phase 3: Gap 분석 중 수작업

### P3-1. 서비스 간 교차 검증 `[교차검증]`
- API 계약 일치, 이벤트 스키마 일치, 용어 일치 여부 검증
- 불일치 항목은 `cross-service-inconsistency.md`에 기록

### P3-2. 기존 요구사항-코드 매핑 확정 `[문서수집]` `[확정]`
- P0-3에서 수집한 기존 문서 기반으로 요구사항↔코드 매핑

### P3-3. 개인정보보호법 적정성 판단 `[인터뷰]`
- 법무/개인정보보호 담당자가 수집 동의, 보관 기간, 파기 절차 검토

### P3-4. 보안 취약점 대응 우선순위 결정 `[확정]`
- 보안센터가 Gap 항목별 대응 우선순위 결정

### P3-5. 서비스 합리화 판단 `[확정]`
- 기능 중복/저사용/고비용 서비스 식별 → 통합/폐기/재작성 후보 판단
