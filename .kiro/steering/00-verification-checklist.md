# 설계도 완전성 검증 체크리스트 (매 작업 시 자동 적용)

> 이 steering은 `inclusion: auto`로 설정되어 모든 대화에 자동 로드됩니다.
> KIRO가 `services/*/docs/` 하위에 산출물을 생성·수정할 때마다 아래 검증을 수행하세요.
> 작업자가 매번 요청하지 않아도 KIRO가 자동으로 체크합니다.

---

## A. 산출물 생성 시 필수 검증 (매 Step 완료 시)

### A-1. 코드 기반 완전성 검증
- [ ] 2-Pass 스캔을 수행했는가? (Pass 1: 구조 스캔 → Pass 2: 정밀 분석)
- [ ] ❌ 판정 전 6회 검색 최소 검증을 수행했는가? (grepSearch 3회 + fileSearch 2회 + 설정파일 1회)
- [ ] 누락 검증 로그 테이블을 산출물 하단에 포함했는가?
- [ ] 멀티 모듈(3개+) 서비스에서 모듈별 분할 분석을 수행했는가?

### A-2. 사용자 추가 정보 반영 검증
- [ ] `runtime-data.md` (수작업 수집 런타임 데이터)가 존재하면 해당 내용을 산출물에 반영했는가?
- [ ] 작업자가 채팅에서 추가로 알려준 정보(인터뷰 결과, 운영 환경 등)를 산출물에 반영했는가?
- [ ] 반영한 추가 정보에 `[작업자 제공]` 또는 `[인터뷰 결과]` 태그를 붙였는가?
- [ ] `glossary.md` 용어 사전의 정규화된 명칭을 사용했는가?

### A-3. Step 간 교차 검증
- [ ] 이전 Step 산출물과 ID/명칭/경로 참조가 일치하는가?
- [ ] 교차 검증 로그 테이블을 산출물 하단에 포함했는가?
- [ ] 불일치 발견 시 이전 산출물을 보정하고 `[교차검증 보정]` 태그를 붙였는가?

### A-4. 설계도 양식 완전성 검증
- [ ] 모든 항목에 확인 상태(✅/⚠️/❌/🔍)를 표기했는가?
- [ ] 모든 항목에 분석 주체 태그([KIRO]/[KIRO+도구]/[KIRO+확정]/[인터뷰])를 표기했는가?

### A-5. 완전성 자가진단 리포트 (매 Step 산출물 하단 필수 포함)

> 비개발자 작업자가 산출물 품질을 판단하기 어려운 문제를 해결하기 위해,
> KIRO가 매 Step 완료 시 산출물 하단에 아래 형식의 자가진단 리포트를 자동 생성한다.
> 작업자는 신뢰도 등급과 추가 검증 프롬프트만 확인하면 된다.

#### 자가진단 리포트 형식 (산출물 하단에 삽입)

```markdown
---
## 🔍 완전성 자가진단 리포트

### 분석 커버리지
| 지표 | 값 | 판정 |
|---|---|---|
| 분석 대상 파일 수 (Pass 1 스캔) | {N}개 | — |
| 실제 읽은 파일 수 (Pass 2 정밀) | {M}개 | {✅ 60%+ / ⚠️ 40~60% / 🔴 40% 미만} |
| ❌ 해당없음 판정 항목 수 | {X}개 | {✅ 3개 이하 / ⚠️ 4~7개 / 🔴 8개+} |
| 🔍 인터뷰필요 판정 항목 수 | {Y}개 | — (참고) |
| 항목 수 3개 이하 카테고리 수 | {Z}개 | {✅ 0개 / ⚠️ 1~2개 / 🔴 3개+} |
| 컨텍스트 윈도우 리셋 횟수 | {R}회 | {✅ 0회 / ⚠️ 1회 / 🔴 2회+} |

### 신뢰도 등급

| 등급 | 조건 (하나라도 해당 시 해당 등급) |
|---|---|
| 🟢 HIGH | 파일 커버리지 60%+, ❌ 3개 이하, 빈약 카테고리 0개, 리셋 0회 |
| 🟡 MEDIUM | 파일 커버리지 40~60%, 또는 ❌ 4~7개, 또는 빈약 카테고리 1~2개, 또는 리셋 1회 |
| 🔴 LOW | 파일 커버리지 40% 미만, 또는 ❌ 8개+, 또는 빈약 카테고리 3개+, 또는 리셋 2회+ |

> **이 산출물의 신뢰도: {🟢/🟡/🔴} {등급명}**

### 🟡/🔴 시 추가 검증 프롬프트 (복사해서 채팅에 붙여넣기)

> 아래 프롬프트는 KIRO가 자동 생성합니다. 작업자는 해당되는 프롬프트를 복사하여 채팅에 붙여넣기만 하면 됩니다.

1. (❌ 해당없음 재검증) `❌ 해당없음으로 판정된 {X}개 항목을 재검증해줘. 특히 {누락 의심 카테고리 나열}.`
2. (미분석 파일 추가 스캔) `분석하지 못한 파일 {N-M}개 중 주요 파일을 추가 스캔해줘. 특히 {미분석 디렉토리 나열}.`
3. (빈약 카테고리 집중 분석) `이 산출물에서 항목 수가 3개 이하인 카테고리({빈약 카테고리 나열})를 집중 재분석해줘.`
4. (컨텍스트 리셋 보완) `이전 세션에서 분석한 내용과 이번 산출물을 교차 확인해줘. 누락된 항목이 있으면 보완해줘.`
```

#### 자가진단 리포트 생성 규칙

1. **매 Step 산출물 생성 시 필수**: `services/*/docs/` 하위에 산출물을 생성할 때마다 해당 산출물 파일 하단에 자가진단 리포트를 포함하라
2. **지표 수집 방법**:
   - "분석 대상 파일 수": Pass 1에서 `listDirectory` + `fileSearch`로 식별한 파일 수
   - "실제 읽은 파일 수": Pass 2에서 `readCode`/`readFile`로 실제 내용을 읽은 파일 수
   - "❌ 해당없음 판정 수": 산출물 내 ❌ 표기 항목 수
   - "항목 수 3개 이하 카테고리": 산출물 내 각 섹션/카테고리별 항목 수를 세어 3개 이하인 것
   - "컨텍스트 윈도우 리셋 횟수": 대화 중 새 세션으로 전환된 횟수 (0이면 단일 세션 완료)
3. **신뢰도 등급 산정**: 위 조건 테이블에 따라 가장 낮은 등급을 최종 등급으로 채택
4. **추가 검증 프롬프트 자동 생성**: 🟡 또는 🔴 등급일 때, 해당 지표에 맞는 프롬프트를 자동 생성하여 작업자가 복사-붙여넣기만 하면 되도록 구체적으로 작성하라
   - 프롬프트에는 구체적인 카테고리명, 디렉토리명, 항목명을 포함하라 (추상적 표현 금지)
5. **🟢 HIGH일 때**: 추가 검증 프롬프트 섹션에 "✅ 신뢰도 HIGH — 추가 검증 불필요. 다음 Step으로 진행하세요." 표기
6. **🔴 LOW일 때**: 추가 검증 프롬프트 앞에 아래 경고를 표시하라:
   ```
   ⚠️ 신뢰도 LOW — 이 산출물은 누락이 있을 가능성이 높습니다.
   아래 프롬프트를 순서대로 실행하여 보완한 후 다음 Step으로 진행하세요.
   보완 후에도 LOW가 유지되면, 모듈/범위를 나누어 Step을 재실행하세요.
   ```
